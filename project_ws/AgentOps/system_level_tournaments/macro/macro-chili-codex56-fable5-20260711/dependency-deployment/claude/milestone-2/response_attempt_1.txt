Design decisions, briefly: the m1 test pins `execute_deployment(services)["status"] == "planned"`, so a call with no `deploy`/`healthy`/`rollback` callbacks stays a plan-only run (preserving m1 behavior exactly, including `state.plan`). Any callback provided switches to execution: deploy each wave in wave order, health-check every member of the wave (no short-circuit — the whole wave is checked), and on any unhealthy service roll back all deployed services in strict reverse deployment order, returning `"rolled_back"`; a clean run returns `"deployed"`. The result always carries the four contract keys (`status`, `waves`, `deployed`, `rolled_back`), `deployed` is the full deploy-order list so `rolled_back` is its exact reverse, and `DeploymentState`'s existing `record_deployed`/`record_failed`/`record_rolled_back` hooks are now driven by the executor when a state is passed. Only `app/executor.py` changes; `model.py`, `graph.py`, and `state.py` already satisfy their contracts and stay untouched.
