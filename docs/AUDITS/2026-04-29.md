# CHILI Trading-Brain Comprehensive Audit — 2026-04-29

Comprehensive third-pass audit following the 2026-04-28 day-of-fix sprint
(see [`2026-04-28-deep-fixes.md`](2026-04-28-deep-fixes.md)). Probes hit the
live `chili` Postgres on port 5433 directly via `psycopg2` from the audit
sandbox; the `_claude_pending.txt` dispatch route was bypassed because a
prior PowerShell `DO $$ ... $$` heredoc wedged the daemon (see Finding D-13).

Two priors govern the recommendations:

1. **Hard Rule 4 / data-first**: when the bug looks like contaminated linkage
   or wrong unit, fix the DB plus migration; do not paper over with router
   filters.
2. **No hardcoded fallback values** (operator feedback, 2026-04-29):
   wherever the current code uses `value or 0.5` / `default=0.5` /
   `confidence = 0.5  # default`, the audit calls it out and proposes a
   dynamically computed replacement (population mean, Bayesian posterior,
   or `None`-propagation). The 0.5 magic number is treated as a bug pattern,
   not just a code-smell.

#### A-1 — PDT designation rejecting 1,333 exits / 24h `[CRITICAL]`

WHERE: `app/services/trading/auto_trader.py` (entry funnel) +
`app/services/trading/auto_trader_monitor.py` (exit funnel) +
`app/services/trading/venue/robinhood_spot.py`

EVIDENCE (live probe 2026-04-29):
```
decision               | reason                            | n
monitor_exit_rejected  | Sell may cause PDT designation.   | 1333
monitor_exit_rejected  | Not enough shares to sell.        |   15
monitor_exit_rejected  | JSON parse error - Expecting...   |    1
```

IMPACT: Every monitor pass attempts to close positions Robinhood will
refuse for PDT-protection reasons. The system burns ~55 broker calls/hour
on guaranteed-fail exits, drowning legitimate broker activity in PDT
errors and producing log-storm.

ROOT CAUSE: PDT is a pre-trade *entry* check, not an exit-time check.
Robinhood blocks the exit because closing today would constitute the
N-th day-trade in 5 days for a sub-$25K margin account. The autotrader
opens positions without consulting the day-trade counter.

FIX (sketch):
```python
# new module: app/services/trading/pdt_guard.py
def can_intraday_round_trip(account_equity: Decimal, day_trades_5d: int) -> bool:
    if account_equity >= Decimal("25000"):
        return True  # not subject to PDT
    return day_trades_5d < 3  # 4th would trigger
```
Wire `pdt_guard.can_intraday_round_trip(...)` into the entry gate before
`broker_service.place_order(...)`. Pull `day_trades_5d` from the
`trading_trades` table by counting today-opened-AND-today-closed in the
last 5 business days. Pull `account_equity` from a fresh broker portfolio
sync (cache TTL ≤ 60s). Per the no-hardcoded-fallback rule:
`day_trades_5d` must be a real query, not a `or 0` default.

RISK: Don't widen the PDT threshold to bypass the rule — it's a regulatory
constraint, not a CHILI policy. Don't disable the broker's protective
rejection; respect it.

#### B-2 — 626/732 patterns (85%) have NULL avg_return_pct `[HIGH]`

WHERE: `app/services/trading/learning.py` (realized PnL update path)

EVIDENCE: `SELECT COUNT(*) FROM scan_patterns WHERE avg_return_pct IS NULL`
→ 626. Of those, many also have `trade_count=0`, but a meaningful share
do have realized trades that simply never propagated to the
`scan_patterns` aggregate.

IMPACT: The EV gate cannot apply (`min_avg_return_pct` requires a
non-null value). Patterns silently bypass the gate by being NULL, not
by being good.

ROOT CAUSE: Realized PnL update path runs on close; if the pattern's
trades have no `pnl` filled (e.g. cancelled-with-NULL-broker_order_id —
see A-2) the aggregate never updates.

FIX: One-time backfill SQL:
```sql
UPDATE scan_patterns sp
SET avg_return_pct = sub.avg_ret,
    win_rate = sub.win_rate,
    trade_count = sub.n
FROM (
    SELECT scan_pattern_id,
           COUNT(*) AS n,
           AVG(((exit_price - entry_price) / NULLIF(entry_price, 0)) * 100.0) AS avg_ret,
           SUM(CASE WHEN pnl > 0 THEN 1 ELSE 0 END)::float / NULLIF(COUNT(*), 0) AS win_rate
    FROM trading_trades
    WHERE status = 'closed' AND scan_pattern_id IS NOT NULL
    GROUP BY scan_pattern_id
) sub
WHERE sp.id = sub.scan_pattern_id;
```
Add a daily job that runs this for any pattern with a closed trade in
the last 24h.

The pattern is consistent: replace `or 0.5` with one of:
1. Propagate `None` and let the caller decide.
2. Compute from observable data — typically `AVG(realized_metric) FROM
   trading_trades WHERE closed_at > NOW() - INTERVAL '90 days'`.
3. Use Beta(α,β) Bayesian shrinkage with `α = wins, β = losses, prior =
   AVG(...)` — never `α=β=1` (which would itself be a hardcoded prior).

Reference fix module to add: `app/services/trading/dynamic_priors.py`:
```python
def population_win_rate(db) -> float | None:
    row = db.execute(text("""
        SELECT AVG(CASE WHEN pnl > 0 THEN 1.0 ELSE 0.0 END)
        FROM trading_trades
        WHERE status='closed'
          AND exit_date > NOW() - INTERVAL '90 days'
          AND pnl IS NOT NULL
    """)).scalar_one_or_none()
    return float(row) if row is not None else None  # never falls back
```

Every call site that currently `or 0.5`s should call this and propagate
None if it returns None.
