## Verification

### Subtask 1 — decay_miner watchdog ✅

Container log line on every restart:

```
[INFO] [fast_path] decay_miner watchdog started (interval=60s)
```

Visible in `docker compose logs fast-data-worker --since 12m`. Polls `task.done()` every 60s; `record_error("decay_miner", reason)` on death. No restart policy (per brief — that's a future decision).

Implementation detail: handles `asyncio.InvalidStateError` (the brief race where `task.done()` returns True but `task.exception()` raises before the result is materialised) by continuing the loop rather than crashing. `asyncio.CancelledError` is caught and surfaced via `record_error("decay_miner", "task cancelled")` then the watchdog returns. Plus a top-level `except CancelledError: return` to honor cooperative shutdown when the supervisor cancels the watchdog itself.

## Surprises / deviations

1. **Subtask 2's design needed a follow-on fix to actually work end-to-end.** The brief's pattern (set `_last_error = None` on streak threshold, rely on flush to persist) failed for two reasons:

   - **Cross-restart staleness**: in-memory `PairStatus.last_error` defaults to `None` on restart, so `record_healthy_tick`'s `if ps.last_error is None` early-return fires on every tick — the self-clear logic is a no-op for any error that landed in a prior process. Fix: load existing `last_error` from `fast_path_status` in `register()`. Wrapped in try/except so a startup-time DB hiccup doesn't block tracker registration.
   - **COALESCE upsert semantics**: the existing flush used `last_error = COALESCE(EXCLUDED.last_error, fast_path_status.last_error)`, which preserves the existing DB value whenever the in-memory value is `None`. So even an *in-process* error→recovery clear never propagated. Fix: switched `last_error` to direct `EXCLUDED.last_error` overwrite. Safe because in-memory state now always reflects DB truth (point above).

   Caught these by running the brief's verification SQL after the original commit (`5db52b9`), seeing `last_error` unchanged after 8 minutes of polling, then realizing the in-memory state was None throughout. Follow-on fix is `75c5776` (local; see #2 below).

2. **Host-side network outage during the verification window.** During the latest restart cycle, the host fast-data-worker container lost reachability to Coinbase WS endpoints — `[Errno 101] Network is unreachable`, `[Errno 111] Connect call failed`. This is environment-side, not caused by any commit in this task. Side-effects on verification:
   - The brief's "wait 5 minutes of clean streaming, observe last_error clears" step couldn't complete naturally because no bars arrive when WS is unreachable.
   - The final commit `75c5776` (the follow-on fix above) is committed locally but the `git push` is failing with `getaddrinfo() thread failed to start` (DNS resolution from the host has also broken). Retried once; same failure.
   - The other three commits (`000fbc0`, `5db52b9`, `4b2f0ab`) pushed successfully *before* the outage hit.

   Nothing in this task touched the network stack; the outage timing relative to my workflow is coincidence. Will retry the push once the operator's network recovers — the fix is small, safe, and additive. F8a soak will resume accumulating organic data once WS reconnects.

3. **Watchdog created a synthetic "ticker" called `decay_miner` in the in-memory PairStatus dict** when `record_error("decay_miner", ...)` fires. This is fine — `record_error` registers the ticker via `get(ticker)` which calls `register(ticker)` if missing. The synthetic key never gets a real `last_bar_at`, so the candle_freshness probe ignores it. It does count toward `tickers_tracked` in supervisor stats, off-by-one cosmetic. Could pre-register at supervisor boot if Cowork prefers; currently lazy-on-first-error is fine.

## Open questions for Cowork

1. **The follow-on fix `75c5776` should arguably have been part of `5db52b9`.** I committed them separately because (a) I'd already pushed the original commit when I discovered the gap, and (b) I wanted the git log to show clearly that the original implementation had a bug and was patched. If you'd prefer a clean single commit with the fix amended in, I can squash and force-push when the network recovers — but standard practice in this repo so far has been "new commit, not amend." Leaving as-is unless you say otherwise.

2. **The `register()` DB-read on first registration.** Adds one SELECT per ticker per process boot. With 5 tickers that's 5 round-trips at startup. Negligible cost. Worth flagging because it's the only DB read in `StatusTracker`'s hot path; everything else is in-memory.

3. **Watchdog interval of 60s.** Means a silently-failed decay_miner stays unannounced for up to 60s. Could tighten to 10s if Cowork prefers tighter detection; cost is one task `done()` poll every 10s instead of 60s. Both are negligible. 60s aligns with the existing supervisor metrics tick rate.

4. **Validating the watchdog works end-to-end** (not just startup line) requires deliberately killing the decay_miner task in a test harness. Not done in this task — would have required adding a test endpoint or REPL hook to externally cancel the task. Synthetic verification of the inner logic (the `record_error` call path) was done; the structural wiring (task lifecycle observation) is verified via the visible startup log but not by an actual death-and-recover cycle.
