### Construction site:

`app/services/trading/venue/coinbase_spot.py:900-934` (`_handle_l2`): every Coinbase L2 update event creates fresh `BookLevel` objects (up to 20 bids + 20 asks per snapshot) wrapped in a `BookSnapshot`, then calls `buf.update(snap)`. At observed Coinbase emit cadence (~80 snapshots/min total across 5 pairs) that's ~3,200 BookLevel/min created.

The `OrderBookBuffer` prunes by age (`_max_age = 120s`). At steady state the buffer holds 120s × 80 snap/min = 160 snapshots × ~40 BookLevels = ~6,400 BookLevels per pair, ~32k total — matches the observed plateau.

**This is bounded, not leaking.** The prune logic is correct.

### Subtask 3 — surgical fix ✅

Tightened `_DEFAULT_BOOK_HISTORY` from 120s → 30s and `_DEFAULT_TRADE_HISTORY` from 300s → 90s in `app/services/trading/microstructure.py`. Justification: only consumer of the global book buffer is `compute_features()` which calls `buf.latest()` (one snapshot, age irrelevant) and never reads `recent()`; the trade-buffer consumer asks for `trade_window_secs=60.0` so 90s is 1.5x headroom.

## Surprises / deviations

1. **The 30-min window did not reproduce the operator's 1 GiB / 2.3h cycle.** Pre-fix slope in this window was 46 MB/hr; the operator's pre-WSL2-cap symptom implies ~435 MB/hr. **A factor of ~10 between observation and reported symptom.** Possible explanations:
   - Event-driven leak: a specific endpoint or scheduler-job triggers the burst. With the operator not actively using chili during my window, the trigger never fired.
   - The WSL2 cap (f-leak-1.2, applied between sessions) somehow reduced the slope to current levels — though structurally that shouldn't be possible since chili's own 3 GiB cap was unchanged.
   - The pre-cap symptom was a transient (a specific period of high activity, maybe a backtest or import burst, that's no longer running).

   This is the brief's branch C: I cannot match success criterion #4 in a single session because I cannot reproduce the slope. The mem_watcher is now in place; the next session has the right diagnostic.

2. **`vm_rss` is far less reactive than object counts.** Pre-fix BookLevel grew 30k objects but vm_rss only grew 23 MB. Each BookLevel dataclass is ~150-200 bytes. The OS page allocator / Python's free-list keeps the RSS high even when objects are GC'd, so vm_rss numbers are sticky relative to actual heap pressure. **Object counts are the right primary metric**, not RSS.

3. **scheduler-worker's `_make_invoke_excepthook` survivors** (mentioned in f-leak-1's CC report as growing +14/min) ARE present in chili too but stable, not growing. So it's not a chili-side leak; whatever creates Threads in scheduler isn't on chili's hot path.

4. **The chili-brain-worker idle-in-tx** that f-leak-1's postgres integrity probe found: still being auto-handled by `db_watchdog`. Not a memory leak; not in scope here.
