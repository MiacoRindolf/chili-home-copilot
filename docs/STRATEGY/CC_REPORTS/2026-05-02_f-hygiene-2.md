### Subtask 1 — `db_errors = 13` diagnosis ✅ → real bug → fixed

**Investigation grep (sanitized):**

```
27 ×  decay_miner dispatch failed channel=fp_exit_inserted:
        Multiple rows were found when one or none was required
 2 ×  decay_miner dispatch failed channel=fp_exit_inserted:
        (psycopg2.errors.UndefinedColumn) column e.source_alert_id does not exist
   …  decay_miner alert <ID> ... no entry price (best_ask=None ...)
        ← these are pre-fix history (F8a's cross-ticker drain bug, F6.3's
          bar-features case); both fixed in earlier commits and not contributing
          to current db_errors growth
```

**Diagnosis category: B — real bug in the miner's exit-validation query.**

The exit-validation `SELECT` in `_handle_exit_inserted` joined `fast_executions` to `fast_alerts` on `(ticker, alert_type, fired_at)` and called `.one_or_none()` with **no `LIMIT`**. Snapshot-replay catchup at restart can fire several `volume_breakout_pullback_long` alerts with identical `fired_at` to the microsecond — the F8a deferred-emit heap drains a burst on a single book emit, all stamped with the same `now()`. When an exit later references one of those entries, the join returns multiple rows and SQLAlchemy raises `MultipleResultsFound`.

The same join in `exit_manager._fetch_source_alert_meta` already had `LIMIT 1`. The decay_miner copy missed it.

**Surgical fix** (commit `742394f`): added `ORDER BY a.id DESC LIMIT 1` and switched to `.first()`. Most-recent-id wins is fine — duplicates are functionally identical (same source bar produced them all). Same exit-validation invariant that was already encoded in the exit_manager.

**Pre-fix vs post-fix:**

```
PRE-FIX  (last hour before restart):  errs=13  (frozen, not growing)
POST-FIX (60s after restart):          errs=0
```

The 13 was frozen from prior catchup-driven storms; no new `MultipleResultsFound` events have fired since the fix landed. The dispatch-decay-heap-trend output (subtask 3) confirms `errs=0` across every metrics tick after the restart.

## Surprises / deviations

1. **Subtask 1 was decision branch B, not A or C.** The brief framed branch A as the most likely outcome ("known-transient psycopg2 noise, document it") and B as a possibility. The actual finding was a missed `LIMIT 1` in the decay_miner copy of a join that already had `LIMIT 1` in the exit_manager. Concrete fix at the actual error site, no scope creep.

2. **Two distinct error categories in the 24h grep window, both already-fixed-class.** The 27 `MultipleResultsFound` were the live source of growth (now fixed). The 2 `column e.source_alert_id does not exist` were from a transient state during F6.3 development (since fixed). The `no entry price` warnings predate F8a-fix's per-ticker heap (also already fixed). All other categories not contributing to current `db_errors` count.

3. **The frozen-at-13 hypothesis (decision branch C) was technically correct as a description of the snapshot,** but the underlying mechanism (catchup-driven duplicate fired_at) means the counter would tick up again on the next snapshot-replay storm. So branch C was a partial diagnosis; branch B is the durable one.

4. **The dispatch script's `--since` argument behavior:** `docker compose logs --since 6h` actually starts ~6h before the *log line that's currently being written*, not 6h before now. So the first line is 6h-and-some-seconds-old, and the tail is current. This is fine for the diagnostic question but worth flagging if anyone tries to use the script for a precise window.

## Open questions for Cowork

1. **The `MultipleResultsFound` could in principle still happen on legitimate same-microsecond firings,** even outside catchup. Two genuine alerts with identical `(ticker, alert_type, fired_at)` to the microsecond would produce two `fast_alerts` rows; my fix's `ORDER BY a.id DESC LIMIT 1` resolves it deterministically (most recent wins). For exit-validation purposes this is fine because the duplicates are functionally identical; for a future feature that distinguishes between them, a different deduplication strategy might be needed. Not on today's roadmap; flagging in case it becomes relevant.

2. **The dispatch script's `--since` semantics drift** — Docker treats it as relative to log-write time, not wall-clock. Could add a `Get-Date` filter inside the script to enforce a strict wall-clock window. Not done; the current behavior is fine for "is the trend healthy?" questions but would be misleading for precise interval analysis.

3. **The watchdog OK heartbeat adds ~60 lines/hour to the supervisor log volume** (~1440/day). Insignificant compared to the existing supervisor metrics tick (~60 lines/hour at 1-line each, plus per-pair status — already in the hundreds/hour). Worth flagging only if there's log-aggregator cost-per-line concern.
