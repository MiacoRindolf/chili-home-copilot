## Surprises / deviations

1. **Reordered gates contradicting the brief's explicit "AFTER" hint.** The brief said put `gate_negative_edge_excluded` AFTER `gate_calibrated_tradeability`. But the brief's verification SQL — `reject_reason LIKE 'negative_edge%'` — only succeeds if `negative_edge` is the *first* deny on the alert. Under the AFTER ordering, statistically-negative signals (like volume_breakout_long with mean=−27 bps) ALSO fail `is_score_tradeable` (mean far below the +200 bps cost-bar) → tradeability fires first → reject_reason="signal_not_tradeable", shadowing negative_edge. To make the verification work, I swapped the order to negative_edge BEFORE tradeability. The fourth commit (`c2caf80`) did this and explains it inline. Both gates still always run; only the *primary* reject reason changes; gates_json carries every gate's verdict for postmortem.

2. **Volume_breakout_long isn't reaching the negative_edge gate in live traffic.** Most volume_breakout alerts are bar-replay alerts arriving > 60s after their `fired_at` timestamp; `gate_recency` rejects them first with `recency:alert_too_old`. That's the recency gate doing its job — historical bars from snapshot replay shouldn't be acted on. The negative_edge gate IS working (verified directly via the helper, returning EXCLUDED on volume_breakout_long buckets) — it's just shadowed in this dataset by the upstream recency filter. Under a non-replay regime (live feed only, no historical replay), volume_breakout would reach negative_edge.

3. **Floor verification couldn't be done via the brief's SQL on `fast_exits`.** The query expects `fast_exits` rows with `effective_max_hold_s >= 10` and `calibrated_max_hold_s < 10`. But the calibration gates are now blocking nearly every paper fill, so no new entries → no new exits → empty result set. Verified the *logic* directly via the helper (above, success criterion #1). The brain_json field is wired and will populate as soon as a calibrated entry passes through the gates and exits.

4. **DOGE-USD imbalance_long low bucket is statistically negative — even though the magnitude is tiny.** `mean=−0.65 bps, upper_ci=−0.48 bps` with n=124. This is a weird result: such a small mean with such tight confidence interval. Most likely explanation: it's a real but trivially-small structural microbias (maybe a 1-tick adverse move dominates the signal at the 14400s horizon). Worth flagging because the gate is correctly triggering on it, but operators should not interpret "this signal is statistically negative" as "this signal is meaningfully bad" — at 0.5 bps it's basically zero. See Open Question 1.

5. **Effective max_hold floor logic is correct but currently unprovable via the success criterion SQL.** Calibration says some buckets want max_hold=1s. The floor substitutes 10s. But the calibration gates are now (correctly) blocking the fills that would have been governed by that floor — so no fills, no exits, no rows in fast_exits to verify against. This is the gate doing its job and is also a chicken-and-egg verification problem. Direct helper test confirms the substitution; live observation has to wait for either gate-threshold tuning or new signals (F8).

## Open questions for Cowork

1. **DOGE-USD `imbalance_long` low bucket excluded as `negative_edge` — should the rule have a magnitude floor too?** With n=124, the upper-CI is −0.48 bps; the rule correctly says "statistically below zero" but practically "below zero by a vanishingly small margin." Should `is_negative_edge_excluded` require BOTH `upper_ci < 0` AND `|mean| > some_minimum` (say, 1 bp)? Otherwise we'll exclude high-volume buckets with trivially-small biases. Don't tune now; flag for design.

2. **Soak window with calibration gates is producing no fills.** Three calibrated gates (`tradeability`, `negative_edge`, plus the existing `min_score`) plus `recency` are blocking essentially everything under current data. The system is correctly refusing to trade no-edge signals — but that means F6.5's behavioral effect on production realized P/L is "0 fills" until F8 produces a tradeable signal or thresholds get explicitly relaxed. Two implications:
   - Verification of F6.5's downstream effect (floor on max_hold, etc.) requires either F8 first or temporary threshold relaxation. Brief said don't lower thresholds; I followed.
   - The autopilot UI's "Closed Round Trips" view will go quiet (no new closes from new signals) until F8 lands. Operators looking at the page should know the silence is the system working as designed.

3. **The execution latency floor (10s) is unverifiable in live traffic right now** for the same reason as #2. Direct helper test confirms the logic; live observation is gated. If F8 produces a signal that gets through, the floor will start showing up in `effective_max_hold_s`.

4. **`MIN_NEGEDGE_SAMPLES = 30`.** Followed the brief's default. Conservative but defensible — 30 is the conventional t-stat-stability floor. If you have strong reason for 50 (more conservative — fewer false-positive exclusions) or 20 (more aggressive — more exclusions earlier), propose. Not tuning silently.

5. **Volume_breakout_long is shadowed by recency gate in live traffic.** If we want the negative_edge gate to actually catch volume_breakout in production (rather than relying on recency to do it), we'd need to either tighten recency or filter the snapshot-replay path differently. Out of scope per the brief, but worth flagging — the most important signal we wanted to auto-block is currently being blocked by a different gate. Not a problem for safety, but the postmortem reason field doesn't tell the full story unless an operator reads gates_json.

## Verification SQL (verbatim, for next review)

Floor (will populate as F8 produces tradeable signals):

```sql
SELECT entry_execution_id,
       (brain_json->>'calibrated_max_hold_s')::float AS calib,
       (brain_json->>'effective_max_hold_s')::float AS effective,
       brain_json->>'bracket_source' AS src
FROM fast_exits
WHERE entered_at > NOW() - INTERVAL '15 minutes'
ORDER BY entered_at DESC;
```

Negative-edge rejections (currently producing `imbalance_short` until volume_breakout reaches the gate):

```sql
SELECT alert_type, reject_reason, COUNT(*) AS n
FROM fast_executions
WHERE decided_at > NOW() - INTERVAL '15 minutes'
  AND reject_reason LIKE 'negative_edge%'
GROUP BY 1, 2
ORDER BY n DESC;
```

Both gates' verdicts for any single signal type, via gates_json (won't shadow):

```sql
SELECT decided_at, alert_type, decision, reject_reason,
       gates_json->'negative_edge'->>'allow' AS neg_allow,
       gates_json->'negative_edge'->'detail'->>'verdict' AS neg_verdict,
       gates_json->'calibration'->>'allow' AS cost_allow,
       gates_json->'calibration'->'detail'->>'verdict' AS cost_verdict
FROM fast_executions
WHERE alert_type = 'volume_breakout_long'
  AND decided_at > NOW() - INTERVAL '60 minutes'
ORDER BY decided_at DESC LIMIT 10;
```
