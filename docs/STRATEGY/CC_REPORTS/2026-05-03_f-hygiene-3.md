## Open questions for Cowork

1. **Restart-required-to-load-fix tension.** When a fix lands in `app/services/trading/fast_path/*.py`, the only way to load it is to restart fast-data-worker — which interrupts the F8a soak. This task accepted ~30s of interruption. If the F8a soak ever needs to be truly continuous (e.g., multi-day live verification), we'd need a hot-reload mechanism or to time fixes during planned restart windows. Not blocking today; flagging for awareness.

2. **The cold-start backfill aggregates dup alerts as separate observations.** Each catchup-batch dup contributes a separate row to `forward_returns` with identical numerator/denominator. The Welford mean is unchanged (5 identical values × 5 = 5× mean of 1) but VAR_POP becomes artificially small (zero, when all values are identical). That nudges calibration's stderr-based gates (negative-edge exclusion, tradeability) toward false confidence. Worth a future structural-pass fix? My read: not yet. Once F-hygiene-3.1's UPSERT lands more validation residuals, the validation-residual signal becomes the more-trusted accuracy metric; backfill noise affects mean/stderr but not validation residuals.

3. **`validation_only_cells = 0` is structurally OK** (every exit's chosen horizon happens to be a populated cell), but it's also possible the closest-horizon mapping is too coarse — `min(HORIZONS_S, key=...)` snaps to the nearest of {1, 5, 30, 60, 300, 1800, 3600, 14400}s, which for a 49-min average holding time always picks 3600s. If validations all land at 3600s, the stats at shorter horizons never get realized-residual signal. Worth a future task to also write validations to nearby horizons (with weights) so all horizons accumulate residual data?

4. **F8a-evaluation-rerun-2 should re-run with the corrected n=37 framing** (per f-leak-1's amendment) and the now-correct validation-count growth picture. The next iteration's decision tree is more meaningfully fed by these two corrections than by additional soak time.

## Verbatim verification SQL — for next review

```sql
-- 1. Validation-count distribution (the canary)
SELECT realized_validation_count, COUNT(*) AS cells
FROM fast_signal_decay
WHERE alert_type='volume_breakout_pullback_long'
GROUP BY realized_validation_count
ORDER BY 1;

-- 2. Total validations vs distinct exits (drop ratio)
SELECT
  (SELECT SUM(realized_validation_count)
   FROM fast_signal_decay
   WHERE alert_type='volume_breakout_pullback_long') AS total_validations,
  (SELECT COUNT(*) FILTER (WHERE entry_execution_id IN (
     SELECT e.id FROM fast_executions e
     JOIN fast_alerts a ON a.ticker=e.ticker
                       AND a.alert_type=e.alert_type
                       AND a.fired_at=e.alert_fired_at
     WHERE a.alert_type='volume_breakout_pullback_long'
   )) FROM fast_exits) AS distinct_pullback_exits;

-- 3. Validation-only cells (proves UPSERT INSERT branch fired)
SELECT COUNT(*) AS validation_only_cells
FROM fast_signal_decay
WHERE alert_type='volume_breakout_pullback_long'
  AND sample_count = 0
  AND realized_validation_count > 0;
-- Expected to be > 0 within ~24h once a long-horizon exit lands
-- on a cell with no forward-return observation yet.

-- 4. Aggregate-by-IN sanity (correct: 37; wrong if direct-JOIN: 142+)
SELECT COUNT(*) FILTER (WHERE entry_execution_id IN (
  SELECT e.id FROM fast_executions e
  JOIN fast_alerts a ON a.ticker=e.ticker
                    AND a.alert_type=e.alert_type
                    AND a.fired_at=e.alert_fired_at
  WHERE a.alert_type='volume_breakout_pullback_long'
)) AS pullback_exits FROM fast_exits;
```
