## Important framing correction

Before the diagnosis, I checked what `realized_validation_residual` actually measures (commit history showed it at f-hygiene-3.1):

```python
realized_validation_residual = Welford running mean of ABS(realized_return - miner_mean_return)
```

**It's mean absolute error, not signed offset.** The brief's framing — "the miner is underpredicting realized return" — is unsupported by the residual column alone (it's direction-blind). To know direction I'd have to recompute signed errors per-cell from raw exit data.

The diagnosis stands regardless of direction; magnitudes (13-40 bps disagreement) are real signals. But the report below names magnitudes, not signs, where the residual is the only data source.

## Deferred — for f-hygiene-5

The structural fix for hypothesis B. Three candidate approaches:

1. **Per-exit forward-return:** when an exit lands, look up `fast_orderbook` at `fired_at + actual_holding_period_s` (not `fired_at + horizon_s`) and compute realized-equivalent forward-return for the validation residual. Time-aligns miner-mean and realized-return.

2. **Continuous forward-return curve:** decay_miner stores forward-return at multiple sub-horizon timestamps, then validation residual interpolates to actual hold time. More expensive (more obs per alert) but more accurate.

3. **Reframe residual semantics:** the "horizon" in the miner is the *measurement* timestamp, not the *prediction*. Validation residual should always be hold-time-matched, not horizon-matched.

My read: option 1 is the cleanest. ~50-100 LOC. Should fold into a future hygiene brief.

```sql
       ROUND(realized_validation_residual::numeric * 10000, 2) AS resid_bps,

-- 2. Hypothesis B (still relevant for f-hygiene-5)
WITH pullback_exits AS (
  SELECT x.holding_period_s FROM fast_exits x
  WHERE x.entry_execution_id IN (
    SELECT e.id FROM fast_executions e
    JOIN fast_alerts a ON a.ticker=e.ticker AND a.alert_type=e.alert_type AND a.fired_at=e.alert_fired_at
    WHERE a.alert_type='volume_breakout_pullback_long'
  )
)
SELECT
  CASE
    WHEN holding_period_s < 100 THEN '<100s'
    WHEN holding_period_s < 600 THEN '100-600s'
    WHEN holding_period_s < 2700 THEN '600-2700s'
    WHEN holding_period_s < 5400 THEN '2700-5400s'
    ELSE '>5400s'
  END AS hold_bucket,
  COUNT(*) AS exits,
  ROUND(MIN(holding_period_s)::numeric, 0) AS min_s,
  ROUND(AVG(holding_period_s)::numeric, 0) AS avg_s,
  ROUND(MAX(holding_period_s)::numeric, 0) AS max_s
FROM pullback_exits GROUP BY hold_bucket ORDER BY MIN(holding_period_s);
```
