## Findings

### The slope correction is the load-bearing detail

Pre-fix: 178/min. The brief assumed ~50/min from a single 06:19 tick.
The leak was actively accelerating between sessions — likely a
combination of:

- **yfinance's internal state growing over time**, making each failed
  call increasingly expensive (more bookkeeping per Thread spawn).
- **Autotrader sweep cadence** picking up more tickers as the
  pattern-imminent set grew.

The fix caps the leak at the source regardless of arrival rate — the
shared session collapses N concurrent connection attempts into one
pooled client, and the breaker stops attempts entirely after 10
consecutive failures. So the slope-acceleration concern is moot now:
even if yfinance recovers tomorrow and a sudden burst of successful
calls fires, no per-call Thread will be created.

### The breaker is doing exactly the work it was designed for

Five `OPEN` cycles in 6 minutes during sustained yahoo egress block:

```
07:28:26 OPEN: 10 consecutive upstream failures   (initial trip)
07:30:02 HALF_OPEN: probing
07:30:03 OPEN: 11 consecutive upstream failures
07:31:09 HALF_OPEN: probing
07:31:09 OPEN: 12 consecutive upstream failures
...
```

Steady cycle: ~60s OPEN → 1 probe Thread → re-OPEN. Under sustained
yahoo outage, leak rate is now **1 Thread/min** instead of N Threads/sec.
That's the multiplier the breaker delivers on top of the shared session.

Worth noting: the breaker's 60s probe cadence becomes the **floor** on
how quickly chili will detect yahoo's recovery. If yahoo comes back at
07:31:30, the breaker won't notice until the 07:32:16 probe — a 46-second
"missed window" in the worst case. That's an acceptable trade for the
leak cap. If we ever need faster recovery detection (we don't), TTL=60s
becomes the parameter to tune.

### The 5G compose bump is now insurance, not load-bearing

Memory limit went 3G → 5G defensively while the fix soaked. Post-fix
RSS is **flat at ~520 MB** — well under both the old 3G and the new 5G.
The bump can roll back to 3G after a 24h+ clean window per the comment
Claude Code added. Watch item, not action item.

### WinError 10055 is the host-side fingerprint, will heal naturally

The 1 setup-error during the test run is the OS's non-paged kernel pool
saying "no more socket bindings available." That's the cumulative effect
of the leak's outbound socket churn over the prior session. With the
fix landed and yfinance's runaway thread-creation stopped, the OS will
reclaim handles over time. If it doesn't clear in ~30 min of soak, an
explicit `wsl --shutdown` + Docker restart resets the kernel pool.
Currently a watch item.

## Answers to the Open Questions

### 1. Threshold tuning (N=10, TTL=60s)

**Defer.** These are seed values, and the post-fix observation showed
them working — 5 trips in 6 min with no flap, no oscillation. If we see
rapid open/closed cycling once yahoo recovers, that's the empirical
signal to retune. Until then, leave them. Tracking `[yf_breaker] OPEN/CLOSED`
flap rate over the next 24h is the right monitor.

### 2. Reset-on-cache-hit semantics

**Don't change.** Current design (cache hits don't tick the breaker
either way) is correct: cache hits aren't evidence about upstream
health, only about local TTL behavior. Counting them as "success" would
mask actual upstream outages whenever the cache happened to be warm.

### 3. Curl_cffi impersonation profile (`chrome`)

**Watch item.** If we see a 403-storm post-fix where we previously saw
connection failures, that's evidence yahoo's CDN bumped allowed
profiles. The fix is a one-line change to `curl_cffi.requests.Session(impersonate="chrome131"|"safari")`.
Not blocking; just keep an eye on the failure-mode pattern.

### 4. Scheduler-worker parallel drop

**Observe, don't act yet.** The brief noted scheduler had its own
`_make_invoke_excepthook` growth (~+14/min per f-leak-1's report).
Scheduler does import `yf_session`, so if scheduler's leak was the same
yfinance Thread closure, this fix covers both processes. If scheduler's
remaining `_make_invoke_excepthook` growth drops to zero in the next
24h, one fix covered both. If it doesn't, separate brief.
