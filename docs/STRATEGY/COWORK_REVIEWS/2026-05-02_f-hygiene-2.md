## Findings

### The MultipleResultsFound is a structural code-twin issue worth flagging forward

`exit_manager._fetch_source_alert_meta` and `decay_miner._handle_exit_inserted` both look up the source alert for a given exit. Both join the same tables. One had `LIMIT 1`; the other didn't. The identical query exists in two places, and the decay_miner version had the bug.

The narrow fix is correct (add `ORDER BY id DESC LIMIT 1`, `.first()`). The broader question is whether this lookup should be **a single shared helper** so the two sites can't drift again. That's a refactor, not a hygiene-pass item — flagging for a future opportunistic moment, not now.

### Validation density may improve post-fix

Pre-fix, the `MultipleResultsFound` exception in the decay_miner's exit-validation path silently dropped the validation residual update for any exit referencing duplicate-fired_at alerts. Post-fix, those updates land. This could marginally improve `realized_validation_count` on `fast_signal_decay` rows over the next 24h — though the bigger constraint (gate stack blocking pullback fills, no `fast_exits` rows referencing pullback alerts in the first place) is unchanged. So the F8a-evaluation-rerun's "validation residuals are structurally unavailable" finding still mostly holds; this fix just stops corrupting the validation that was already supposed to fire.

### The dispatch script confirms heap health

`pending_heap` oscillates in 480–520 range over 6h. Not growing. F8a-evaluation Open Question 4 closed.

## Answers to the Open Questions

### 1. `ORDER BY id DESC LIMIT 1` for exit-validation — appropriate, or should we deduplicate differently?

**Appropriate. Most-recent-id wins is the right tiebreaker for this use case.**

Same-microsecond duplicates from the catchup heap are *functionally identical* (same source bar, same features, just multiple deferred-emit firings stamped at the same `now()`). Validating against any one of them is equivalent. Using the highest id matches the convention in `exit_manager._fetch_source_alert_meta` — consistency between the two code-twin sites is more valuable than perfect-tiebreaker-theory.

If we ever build a feature that genuinely distinguishes between same-microsecond alerts (e.g., per-emit metadata that differs), this convention would need re-examination. Not on the roadmap. Acknowledged.

### 2. Dispatch script `--since` wall-clock drift

**Acceptable as-is for the current use case. Don't fix.**

The script answers "is the trend healthy?" — which doesn't require precise wall-clock alignment. If we later need it for "what happened between 02:00 and 03:00 UTC?", add a `Get-Date` filter then. Premature now.

### 3. Watchdog OK heartbeat log volume

**Insignificant.** ~60/hour vs the existing supervisor's ~hundreds/hour. Approved.

## Engineering concerns (smaller)

1. **Code-twin divergence between exit_manager and decay_miner** for the `(ticker, alert_type, fired_at) → fast_alerts` lookup. Fold a shared helper into a future opportunistic refactor — *not* a dedicated task. The bug was caught; the divergence is now cosmetic.

2. **Counter reset on supervisor restart** (deferred per brief). With the LIMIT 1 fix, the counter shouldn't grow from this code path going forward. If a different error category emerges that resets the urgency, that's its own decision. Leaving deferred.
