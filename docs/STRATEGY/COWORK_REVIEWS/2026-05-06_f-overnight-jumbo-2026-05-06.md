## What Claude Code did right

1. **Phase 1 correction was the sharpest finding of the run.** The
   brief assumed `tick_normalizer.normalize_price` didn't exist and
   wrote a `_round_stop_price_to_tick` helper from scratch. CC found
   `normalize_price` ALREADY exists and is ALREADY wired at the call
   site (`broker_service.py:2821`). It rounds PED's `13.6275 → 13.63`
   per NMS Rule 612 (equity ≥ $1 = 2-decimal tick).

   So the bug isn't that rounding is missing — it's that the rounded
   price is reaching the wire AND Robinhood is rejecting anyway. CC
   shipped the correct minimal change: add INFO-level pre-submit log
   + WARNING-level full diagnostic on rejection (with `trigger_in`
   AND `normalized_stop` so the rounding delta is visible). **Once
   the next sweep hits PED, the cause becomes unambiguous.** No
   speculative fix; observability first.

   **Brief-cookbook lesson**: this is the third time today's briefs
   missed an existing helper (paper_book_json was wired, normalize_price
   was wired, the 5 handlers were imports-broken-not-missing). The
   cookbook entry "verify saved-memory claims against current code"
   needs to extend to "search for existing helpers before assuming
   you need a new one."

2. **Phase 2's blanket fix was the right judgment call.** Without
   runtime profiling to pinpoint which pydantic model triggers
   per-request rebuild, the surgical fix (Step 2.2) wasn't possible.
   CC shipped Step 2.3's blanket — walk `app.schemas.*` + `app.models.*`
   at startup, call `model_rebuild()` on every BaseModel. **77 models
   rebuilt at startup.** Per-model failures swallowed (one broken
   model can't break startup). Outer try/except in lifespan so even
   if rebuild itself raises, startup completes.

   Trade-off: slightly longer startup time vs eliminating the
   per-request rebuild leak entirely. Right trade for the data
   available.

3. **Source-text test guards as a legitimate test category.** Several
   phases shipped tests that pin source-level invariants (wiring,
   log lines, import patterns, dispatch order) instead of full
   DB-integration tests. <1s per test file vs 7-10 min for
   handler-touching DB tests under the per-test truncate cycle.
   Same regression coverage for accidental wiring deletion at a
   tiny fraction of runtime cost. **This is a precedent worth
   formalizing in the cookbook.**

4. **Anti-pattern grep checks** for the import-bug class. `assert
   "from ....db" not in src` for any new module at depth-5+. Cheap,
   catches the entire class. The 6-day silent regression from
   2026-04-29 to 2026-05-05 wouldn't have been silent if these
   checks had been in place. They are now.

5. **`cron_jobs/` vs `brain_work/handlers/` distinction.** Phases 4,
   5, 9 all created `cron_jobs/*` modules. Phases 3, 6 added to
   `brain_work/handlers/`. Distinction: cron when the natural
   trigger is "every N hours" (broad-market state, periodic cleanup,
   observability snapshots); event-handler when "this thing
   happened" (per-trade-close, per-resolved-alert). The dichotomy
   is explicit in module docstrings going forward.

6. **Phase 6's dispatch order — handlers AFTER demote.** Both
   live_drift and execution_robustness are observability, not
   lifecycle gates. Dispatched after demote so the EV-gate has
   already run. Correctness ordering preserved: pattern_stats →
   demote → live_drift → execution_robustness → regime_ledger.

7. **Phase 8's full file deletion.** All `learning_cycle_report.py`
   helpers were internal-only; CC deleted the whole file rather
   than leaving dead helpers around. Updated
   `learning_cycle_architecture.py` to drop the dead `CycleStepDef`.
   **Right kind of cleanup**: don't leave half-deleted vestiges.

8. **Phase 7's env-var rollback path.** `CHILI_DB_WATCHDOG_BRAIN_WORKER_KILL_SEC=1800`
   restores the 1800s exemption without code change. Defensive
   discipline for a behavioral change.

9. **Phase 3's emitter dedup + per-event try/except.** When emitting
   N events from a sweep, each emit is wrapped so one broken event
   can't poison the rest. Same pattern as Phase 4's stale-promoted
   sweep. Cookbook entry.

### The dispatcher's close-event branch is getting busy

Cross-phase observation #5 from CC: after Phase 6, the trade-close
fanout calls 5 handlers in sequence:

```
pattern_stats → demote → live_drift → execution_robustness → regime_ledger
```

If any handler slows (especially live_drift or execution_robustness,
which do non-trivial refresh work), the entire round slows.
**Watch-item**: post-deploy log grep for >30s per close-event dispatch.
If observed, the right fix is either parallelizing the fanout via
`asyncio.gather` or moving slower handlers to per-handler dedicated
queues.

For now, the sequential dispatch order is correct (pattern_stats
must run first to update inputs that demote reads). Don't parallelize
prematurely.

### Phase 1's diagnostic-first approach is the right discipline for unknown bugs

Brief asked CC to fix PED's stop placement. CC found rounding already
works correctly. Instead of inventing a speculative fix, CC shipped
**only** the diagnostic logging. **The next failed PED sweep will
reveal the actual cause.** This is exactly the right discipline for
"my hypothesis was wrong" situations: ship observability first, fix
based on data, not speculation.

If post-deploy data shows `normalized_stop=13.63` reaching the wire
AND RH still rejecting with the limit-order error, the bug is in
`rh.orders.order`'s internal payload assembly (third-party). That's
a follow-up `f-fix-robin-stocks-payload-assembly` brief.

If the data shows `normalized_stop=13.6275` (rounding bypass), then
the call path is bypassing `tick_normalizer.normalize_price`
somewhere — different fix shape.

Either way, **the next sweep makes the cause unambiguous**, which
is the only honest move when the brief's hypothesis was wrong.

## Answers to the cross-phase observations

### 1. Source-text guards precedent

**Approve as a precedent.** Document in the running cookbook:
"Source-text test guards are a legitimate alternative to full
DB-integration tests when the regression class is 'accidental future
deletion of wiring.' <1s per test file is a 400-600× runtime saving
on this codebase. Use plain-English docstrings explaining the
invariant being pinned."

### 2. Depth-5 absolute imports rule

**Approve as cookbook entry.** Document: "In any
`app/services/X/Y/Z.py` module (depth ≥4 from `app`), prefer
`from app.X import Y` over `from ....X import Y`. The 4-dot relative
form silently resolves to nonexistent `app.services.X` paths in some
package layouts. Anti-pattern grep guard `assert 'from ....db' not
in src` catches the class."

### 3. cron_jobs vs brain_work/handlers

**Approve the dichotomy.** Document: "cron_jobs/ when natural
trigger is 'every N hours' (broad-market state, periodic cleanup,
observability snapshots); brain_work/handlers/ when natural trigger
is 'this thing happened' (per-trade-close, per-resolved-alert,
per-backtest-completion)."

### 4. Per-event try/except in emit batches

**Approve as cookbook entry.** Document: "When emitting N events
from a sweep loop, wrap each emit in try/except so one broken event
can't poison the rest of the batch. Pattern shipped in
f-handler-breakout-outcomes (sweep emitter) and f-cron-stale-promoted
(per-pattern audit-row write)."

### 5. Dispatcher close-event fanout

**Watch item, not action.** 5 handlers in sequence is fine for
current load. If post-deploy logs show >30s dispatch rounds, surface
follow-up brief `f-parallelize-trade-close-fanout`.
