## Findings

### The n=37 correction changes the F8a strategic framing

The original f8a-evaluation-rerun verdict said "142 round trips, avg −6.7 bps, win rate 30%, weakly refuted." With the correction:

- **n=37, not n=142.** The 3.8× inflation came from duplicate `fast_alerts` rows at identical microsecond `fired_at` (catchup pattern; F-hygiene-2.1 fixed this for `decay_miner`'s exit-validation but not retroactively for the report query).
- **Per-ticker means are still directionally correct** — duplicates carry identical values, so the average is the same.
- **But n=37 is suggestive-tier, not verdict-grade** by the protocol's tier definitions. The verdict reframes from "weakly refuted" to "suggestive evidence of weak refutation, awaiting more samples."

The strategic implication: **F9 isn't as obvious a pivot as the prior verdict suggested.** Continued soak on F8a is more strategically informative than I thought. BTC's +4.3 bps subset (still positive, even with corrected n) remains the most interesting per-ticker signal.

### The Thread/Timer closure leak is structural

Scheduler-worker's `mem_watcher` shows `_make_invoke_excepthook.<locals>.invoke_excepthook` growing +14/min. That's exactly one survivor per `Thread()` construction. **Something is creating threads faster than Python's GC reclaims them.** The leak isn't a single cache — it's a request/event handler creating a new Thread (or Timer, since both use the same closure) on each invocation without joining.

Likely candidates worth investigating in f-leak-2:
- Per-request `httpx.Client()` or `requests.Session` (each creates a connection-pool thread)
- `threading.Timer()` for retry/timeout logic that isn't cancelled cleanly
- `concurrent.futures.ThreadPoolExecutor()` instantiated per call instead of per app
- `asyncio.run_in_executor()` with default executor that doesn't get reused

Lifting `mem_watcher` into chili (as Claude Code suggested) gives f-leak-2 the same diagnostic surface scheduler-worker has.

### Chili is currently stable but might be event-driven

7 restarts in the first 16h, then 0 restarts in the past ~8h. The leak might fire on specific user actions or scheduler triggers, not on a steady drip. Subtask 1's continuous logging is exactly the right tool for catching event-driven patterns.
