## What Claude Code did right

1. **Caught its own design gap in subtask 2.** Two real bugs in the originally-shipped commit:
   - In-memory `PairStatus.last_error` defaults to `None` on every process restart, making the `if ps.last_error is None: return` early-return fire on every tick after a restart — the self-clear was a permanent no-op for any error inherited from a prior process. **Most stale `last_error`s in the wild are exactly this category** (an error from a previous session that hasn't been cleared). Without the fix, the feature solves nothing.
   - The flush used `last_error = COALESCE(EXCLUDED.last_error, fast_path_status.last_error)` — a defensive upsert pattern that preserves DB state when the in-memory value is `None`. After the in-memory clear fired, the flush *re-preserved* the old DB error. Self-clear in memory; no propagation. Silent.

   Either of these alone would have rendered the feature inert; together they would have made the failure mode "looks fine in unit-tests, ineffective in prod." Caught by running the SQL the brief asked for, not by trusting the implementation.

2. **Synthetic verification when natural verification was blocked.** Network outage hit during the verification window, so "wait 5 min of clean streaming, observe DB clear" couldn't run live. Claude Code wrote a synthetic harness inside the chili container that exercises every link in the chain (registration, error recording, streak start, threshold crossing, in-memory clear, flush, DB read-back) and confirmed end-to-end. That's the right substitution — verifies the same property the natural test would have.

3. **Honest about what wasn't verified.** Open Question 4 explicitly flags that the watchdog was verified by startup log line + inner-logic synthetic test, but NOT by an actual death-and-recover cycle. That's the gap a future task would close (test harness with externally-cancellable decay_miner). Naming it openly instead of asserting full coverage is exactly the right move.

4. **Restraint on watchdog scope.** Brief said "watchdog reports, doesn't restart." Claude Code held that line. Also resisted the temptation to add watchdogs for scanner / ws_listener / exit_manager opportunistically — kept the change surgical.

5. **`-O`-safe drain change is minimal.** 10 lines added, 9 removed for what is structurally a one-line swap. The diff is what the brief asked for and nothing more.

6. **Acknowledges the cosmetic synthetic-ticker quirk** (Surprise #3) without trying to fix it mid-task. The `decay_miner` lazy-on-first-error in the PairStatus dict is a 1-line cosmetic in `tickers_tracked`. Calling it out for the deferred list is right; not folding a fix into this task is right.

## Findings

### The two `last_error` bugs were structurally similar to F8a's drain bug

Both shipped originally, both verified with the brief's success criteria, both *would have* worked under a narrow interpretation of "does the code path execute" but failed under the broader interpretation of "does the user observe the intended effect."

In F8a, the alert *emitted* correctly with valid lineage; only the price-capture was missing. In `last_error` self-clear, the in-memory clear *fired* correctly; only the persistence to operator-visible DB was missing.

This is a recurring pattern: subsystems that work in isolation but lose information at handoff boundaries (heap-pop ↔ book-emit ticker mismatch; in-memory ↔ DB flush COALESCE). Worth keeping in mind when reviewing future briefs — handoff points are where verification should focus.

### Network outage is a known operational state

The host's WS reachability to Coinbase is gone again — `[Errno 101] Network is unreachable`, `getaddrinfo() thread failed to start`. This is the same recurring egress-block pattern from the audit memory chain (multiple instances since 2026-04-27). Not a regression caused by this task. F8a soak pauses until egress recovers; nothing in this task touched the network stack.

The follow-on commit `75c5776` is local-only because `git push` can't resolve `github.com`. It's safe to push as-is when DNS recovers — small, additive, no migrations.

## Answers to the Open Questions

### 1. Squash `75c5776` into `5db52b9` or leave separate?

**Leave separate. Don't squash, don't force-push.**

The git log already documents the verification-driven discovery cleanly: original commit ships, follow-up commit fixes a gap caught by verification. That's *information*. Squashing erases it. Force-pushing to `main` is also against this repo's pattern. Land `75c5776` as a normal forward commit when the network recovers.

### 2. The 5 SELECTs at process boot in `register()`

Fine. Negligible cost; happens once per startup; no hot-path impact. Worth flagging only because it's the only DB read in StatusTracker — agreed it's worth flagging, but not changing.

### 3. Watchdog interval of 60s

**Fine, leave at 60s.** Aligning with the supervisor metrics tick is the right anchor. A silently-failed decay_miner stays unannounced for up to 60s — the cost of that is "operator dashboard is stale by ≤1 minute," which is well within the protocol's existing latency budget. Tightening to 10s buys nothing meaningful and adds 6× the polling cost. Hold.

### 4. End-to-end watchdog test (death-and-recover)

**Defer to a future task with explicit scope.** The right shape is a small test harness — probably a dev-only endpoint that calls `decay_miner_task.cancel()` followed by polling `fast_path_status` for the synthetic-ticker `last_error`. That's its own brief: "exercise the watchdog death-recovery path with an explicit harness." Not now.

The structural wiring (watchdog observes a real asyncio task) is verified by the visible startup log + the synthetic verification of the inner `record_error` path. The remaining unverified link is *only* "does Python's asyncio Task.done() / Task.exception() actually report what we expect when the wrapped coroutine raises" — and that's stdlib behavior we can rely on.

## Engineering concerns (smaller)

1. **Lazy-registered `decay_miner` synthetic ticker.** Cosmetic, see Surprise #3. If we ever add other watchdogs (scanner, ws_listener), each will also become a synthetic ticker on first error. At that point pre-registering all subsystem-tickers at supervisor boot makes sense; one-at-a-time it's premature.

2. **`register()`'s try/except around the DB read.** Good defensive choice — a startup-time DB hiccup shouldn't block tracker registration. Confirm the `except` logs at least at WARNING so we'd notice if it fires repeatedly. (If not already, fold into next hygiene pass.)

3. **The `last_error` self-clear logs an INFO line on every clear.** Right level. Operator-visible.
