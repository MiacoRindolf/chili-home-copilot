## Findings

### The actual leak source remains unidentified

The 30-min observation window saw normal steady-state behavior at ~46 MB/hr. The operator's symptom requires ~435 MB/hr (1 GiB / 2.3h). Possible explanations, in rough order of likelihood:

1. **Event-driven endpoint hit during a specific period** — e.g., an admin dashboard, a planning page, or an LLM-fallback path that loads large structures into memory and doesn't release them. Would only fire when the operator (or some external client) hits the endpoint.
2. **Scheduler-job-triggered burst** — a periodic learning_cycle or backtest spawn that briefly inflates chili's heap before completing. The 6 AM/PM PT pattern would fit this.
3. **High-volatility period book/trade burst** — at unusually high Coinbase emit rates, even bounded buffers temporarily inflate before the prune catches up. Would explain the WSL2-cap reset solving it (more headroom = less pressure).

Without the next OOM event, we can't distinguish. mem_watcher is the right tool for the next round.

### The buffer-tightening fix is real, even if not the headline

67–70% reduction in BookLevel/TapeTrade/BookSnapshot steady-state object counts. 13 MB stable RSS savings. The 120s book-history default was **3.4× more than the only consumer needed** — over-provisioned by default. Tightening it to 30s is correct on-the-merits, regardless of whether it relates to the operator's OOM cycles.

This is the kind of small structural improvement that compounds: every other piece of code that adds a per-process buffer has now had a precedent for "match the consumer's actual horizon, not the worst case."

### Schedule-worker's per-Thread closure leak is now confirmed cross-process-distinct

f-leak-1's CC report flagged `_make_invoke_excepthook` as a scheduler-side observation; the brief speculated it might also affect chili. Claude Code's mem_watcher run confirms: **on chili it's stable, on scheduler-worker it's growing.** Different leak, different process. Means scheduler-worker has its own leak waiting to be named — but not via this brief.

**Surface as a watch item, not an action item.** With WSL2 cap + buffer tightening + mem_watcher in place, the host is triple-protected. If the OOM cycle recurs, mem_watcher names the type. If it doesn't recur for a week, it was probably a transient operational pattern (specific period of activity that's now over). Either outcome is informative.

## Engineering concerns (smaller)

1. **Buffer-tightening interacts with potential future consumers.** If any future code adds a `recent(window=N)` call where N > 30s for the book buffer, that consumer would see truncated history. The constants are `_DEFAULT_BOOK_HISTORY`/`_DEFAULT_TRADE_HISTORY` — a future contributor adding a longer-window consumer would need to override. Worth a comment near the constants explaining the consumer-side analysis that justified the choice. (Likely already there given Claude Code's discipline; verify in the diff.)

2. **`MemoizedSlots.oneshot.memo` survivors at 2292-2306 (+0.5/min) are dormant** but not necessarily benign forever. If the application surface that creates these grows, they could turn into a slow leak. Watch list, not action list.

3. **scheduler-worker's per-Thread closure pattern remains unfixed.** Different code path; deferred to its own brief if it becomes operationally painful. Not urgent — scheduler has a 10 GiB cap and is currently at 3.32 GiB.
