## What Claude Code did right

1. **Caught a verification-vs-design conflict in the brief and pushed back.** Brief said put `gate_negative_edge_excluded` AFTER `gate_calibrated_tradeability`. But the success criterion was "verify negative_edge appears as a primary reject reason" — under AFTER ordering, statistically-negative signals also fail tradeability (much larger gap from 0 to +200 bps cost bar than from 0 to negative), so tradeability would shadow negative_edge in the postmortem field. Claude Code added a fourth commit (`c2caf80`) reordering them and documented the reasoning inline. Both gates still always run; only the `reject_reason` primary changes. That's the right call: the postmortem reason should be the most informative one, and "negative_edge" is more actionable than "signal_not_tradeable" (the latter is just "edge < cost"; the former tells you the data is conclusive).

2. **Direct-helper verification when the live SQL came back empty.** The brief's verification SQL on `fast_exits.brain_json` came back empty because (correctly) no fills are landing under the new gates. Claude Code didn't fudge — it tested `get_calibrated_max_hold_s()` and `is_negative_edge_excluded()` directly via Python, captured the per-bucket evidence, and surfaced the chicken-and-egg verification problem in Open Questions. Honest test methodology.

3. **Surfaced the surprising-but-correct DOGE finding.** DOGE-USD `imbalance_long` `low` bucket has `mean=−0.65 bps, upper_ci=−0.48 bps` over n=124 — statistically negative but trivially small in magnitude. The gate fires correctly per the rule; Claude Code flagged that the rule may want a magnitude floor (e.g., `|mean| > 1 bp` AND `upper_ci < 0`) to avoid excluding high-sample buckets with vanishingly-small biases. Defensive reasoning.

4. **Volume_breakout_long shadowed by recency gate in this dataset.** Most volume_breakout alerts in current traffic are snapshot-replay from container restart, ages > 60s, getting rejected by `gate_recency` first. So `negative_edge` doesn't fire on them in the postmortem reason field — even though it would, in a non-replay regime. Claude Code surfaced this clearly so we don't read the data wrong.

## Answers to the Open Questions

### 1. DOGE bucket excluded with mean=−0.65 bps — should the rule have a magnitude floor too?

**Yes, eventually.** Add `|mean| > 1 bp` (or maybe `|mean| > stderr * 2` — i.e. the magnitude must be at least one stderr unit away from zero) as a second condition. But not in this task — that's a tuning decision that deserves its own brief. Right now the gate is firing on a bucket with negligible magnitude; it doesn't cost us anything (the alert wouldn't have produced edge anyway), and changing the rule before F8 would be premature optimization.

I'll fold this into a future "calibration tuning" task once F8 produces enough signal-with-actual-edge to make the contrast obvious.

### 2. Soak window producing no fills — is that a problem?

**No, that's the point.** The system is correctly refusing to trade no-edge signals. Three Implications:
- The autopilot UI "Closed Round Trips" panel goes quiet — that's the system working as designed, not a bug. Operators reading the page should know.
- F6.5's downstream effect on realized P/L can't be measured until F8 produces a tradeable signal. Acceptable trade-off; F6.5's purpose was prevention of bad fills, not enablement of new ones.
- Verification of the floor logic in `effective_max_hold_s` waits for the same condition.

If we wanted to validate F6.5's plumbing in production, we'd temporarily relax the trading-cost threshold. We've explicitly chosen not to. **Stay the course.**

### 3. Floor (10s) unverifiable in live traffic right now

Same root cause as #2. Direct helper test confirms the substitution. When F8 produces a signal that survives gating, `effective_max_hold_s` populates and the verification SQL produces rows.

### 4. `MIN_NEGEDGE_SAMPLES = 30`

Hold at 30. The 30-sample floor is the conventional t-stat-stability bar; lowering to 20 risks false-positive exclusions on noisy small samples; raising to 50 means we wait longer to catch genuinely-negative signals. 30 is right.

### 5. Volume_breakout_long shadowed by recency

Real but not urgent. **The system safety property holds either way:** volume_breakout is being rejected, just not via the gate we'd expect to read in postmortem. If we ever want the postmortem reason to reflect the deepest-conceptual-reject (the alert-was-doomed-anyway gate), we'd need to reorder gates so negative_edge runs before recency. But that means slow DB reads happen before the fast in-memory recency check — anti-performance. Better to live with it and read `gates_json` for the full picture.

Operator note: if you ever audit "why didn't volume_breakout fill?" — the surface answer (recency) and the deeper answer (negative_edge) are both correct, just at different layers.

Candidate signal types worth exploring (per F6 review's recommendation set):

1. **Fade volume_breakout** — explicitly trade against the existing signal since it's reliably wrong (-28.5 bps mean over n=120). Cheap experiment: same scanner output, opposite side. If it produces +N bps where the original produced −N bps, we have edge.
2. **Order-book momentum** — rate of change of imbalance over short windows (not raw imbalance). Detects acceleration of one-sided pressure.
3. **Trade tape aggression** — taker-buy ratio over rolling 10s windows from the `market_trades` channel (we'd need to subscribe; we currently don't).
4. **Cross-pair lead-lag** — BTC moves first; alts follow with 1-30s delay. Signal: BTC moves >X bps in T seconds → buy correlated alts that haven't caught up.
5. **Spread compression breakouts (real ones)** — sustained tight spread + volume buildup at a level + breakout. Current `spread_squeeze` is one-tick; needs sustained-state logic.
6. **Time-of-day session signals** — quant lit uses session boundaries; we don't.

The fade-volume_breakout (#1) is the smallest and quickest experiment with the most direct interpretation: "if our existing signal is reliably wrong, the inverse should be reliably right." It's also operationally cheap (no new data subscription, no new feature engineering).

The order-book momentum (#2) and trade tape aggression (#3) require new feature work but are the most quant-credible candidates.

Will discuss with operator before writing the F8 brief.
