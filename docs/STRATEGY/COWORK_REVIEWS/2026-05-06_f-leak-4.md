## Findings

### My brief misattributed which container the leak surfaced in

The pre-fix mem_watcher data showed:
- scheduler-worker: ReferenceType +9-15k per 5min tick
- brain-worker: BookLevel +944 in one tick

I treated those as two separate leaks (Phase 1 and Phase 3
respectively). CC's cross-phase observation suggests they're the
**same leak surfacing in two containers' mem_watchers**. The
strat_cls retention in brain-worker creates BookLevel pandas
weakrefs; those weakrefs are visible in any process that imports
the same SQLAlchemy / pandas modules with the same weakref tracking,
including scheduler-worker.

If post-deploy the scheduler-worker ReferenceType growth drops to
≤1k/5min (per the smoke probe), that confirms the unified-cause
theory. If it stays elevated, there's a separate scheduler-worker
leak that needs its own audit.

**This is a Cowork-side mental-model correction.** I assumed
mem_watcher's per-container data meant per-container causes. CC
recognized that pandas / SQLAlchemy weakrefs cross-pollinate.

### Phase 2 is genuinely separate

The chili main-app +63 MB/min slope is from per-request pydantic
closures, not from the strat_cls leak. Phase 3's fix won't help.
Phase 2's runtime diagnostic needs to run during a heavy-job window
(22:30 UTC cluster) to capture the closure-count growth.

The fix-once-confirmed is small (one or two `Model.model_rebuild()`
calls at module import). But it requires the operator to run the
diagnostic first — Cowork can't pin the offending model from static
analysis alone.

## Brief-cookbook updates from today

Adding to today's running list:

- **Pre-brief leak hypotheses are often wrong about WHERE, even when
  right about SHAPE.** Today's brief identified the leak class
  (SQLAlchemy session lifecycle / pandas retention) but pointed at
  the wrong files. CC's audit found the actual cause in a place
  neither the brief nor I predicted. Going forward: leak briefs
  should phrase hypotheses as "likely candidates" not "the cause,"
  and require the executor to audit ALL recently-shipped sites in
  the leak's class, not just the brief-named ones.

- **mem_watcher per-container data isn't per-container causal.**
  Pandas / SQLAlchemy weakrefs cross-pollinate across containers
  that share host kernel weakref tracking. A leak in container A
  can show up in container B's mem_watcher. Cowork mental model
  needs to account for this. Recommend cross-referencing
  per-container mem_watcher with the actual workload running there
  before attributing leak causes.
